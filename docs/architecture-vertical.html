PodExtract 运行逻辑（竖向）

基于当前实现：API + Redis 队列 + Worker + 分阶段处理 + 飞书回写

入口
1) 用户提交链接：POST /v1/jobs 或 POST /v1/feishu/events
飞书入口先做 token 验证、解析消息文本里的第一个 URL。
↓
来源识别
2) SourceResolver 规范化 URL + 判断平台：YouTube / Bilibili / RSS / Apple Podcast。
Apple 链接会先调 iTunes Lookup 拿 feedUrl，再转成可去重的 canonical URL。
↓
去重
3) SQLite 按 canonical_url + 时间窗口（默认 24h）查重。
↓
命中重复任务：直接返回已有 job_id。
若来自飞书且已有任务在 queued/running，会补写通知目标。
未命中：创建 Job（status=queued）写入 SQLite，
并把 job_id 入 Redis 列表队列。
↓
异步消费
4) Worker 循环 BLPOP 出队 job_id，进入 JobProcessor.process()。
↓
Stage 1: Extract
5) 更新状态为 running/extract，抽取音频。
YouTube/Bilibili：yt-dlp；RSS/Apple：拉 feed，取最新 item 的 enclosure 下载音频。
失败按指数退避重试（默认 3 次），最终失败抛 EXTRACT_FAILED。
↓
Stage 2: Transcribe
6) 更新为 transcribe，调用 STT：主服务默认 MiniMax；失败时切换 ElevenLabs 兜底。
整体按配置重试（默认 2 次），最终失败抛 TRANSCRIBE_FAILED。
↓
Stage 3: Analyze
7) 更新为 analyze，先取 Memory/KB 上下文，再调用 LLM 产出结构化 JSON。
若无 LLM key 或调用异常，走降级结果（保底结构，不中断流程）。
↓
Stage 4: Publish
8) 更新为 publish，写飞书 Doc（raw markdown）+ 可选 Bitable 记录。
失败重试（默认 2 次），最终失败抛 PUBLISH_FAILED。
↓
成功路径：
9) status=completed，stage=done，保存 delivery 信息。
失败路径：
9) status=failed，记录标准化错误码（StageFailure）或 UNHANDLED。
↓
最终通知
10) 无论成功或失败，只要存在 notify_target，都会回发飞书消息（文档链接或错误原因）。
↓
备注：/v1/subscriptions、/v1/digests 目前仅为二期占位接口，不参与一期主处理链路。
